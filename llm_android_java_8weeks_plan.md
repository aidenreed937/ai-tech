
# 从 Android/Java 应用工程师到 LLM 应用工程师  
## —— 8 周高可行性、高性价比落地计划

## 1. 目标与定位

### 1.1 你的当前背景
- 10 年 Android / Java 应用层开发经验  
- 熟悉业务系统、接口设计、数据库、运维  
- 对业务理解强，交付能力稳定

### 1.2 未来定位建议
> **大模型应用工程师 / AI 工程师（RAG + Agent + 自动化）**

不走算法/研究路线，而走 **“业务 + AI” 的落地路线**，核心价值：

- 让大模型真正接入生产环境  
- 用 RAG、Agent、工具编排让流程智能化  
- 保证安全、可靠、可维护、可评估

---

## 2. 高性价比的学习与落地策略

### 2.1 节奏与时间规划
- 每周投入：8–10 小时（工作日晚 + 周末）  
- 总周期：8 周  
- 每周都要有“可以 demo / 可验收”的产物

### 2.2 核心原则：场景优先，而不是技术优先
选一个你最熟的场景贯穿 8 周：

**推荐顺序：**

1. **企业知识助手（强烈推荐）**  
2. 数据报表助手  
3. 工单/客服智能助手

> 别跳来跳去，只优化一个场景，做到能上线试点。

### 2.3 技术路径：先快后稳
#### 第一段（Week 1–2）：快速出样  
使用 **Dify** 或类似平台快速构建：

- 知识库 + RAG  
- 工作流 + 工具节点  
- 可用 Demo（10 分钟搞定）

#### 第二段（Week 3–8）：工程化落地  
- Python（FastAPI） + LlamaIndex/LangGraph  
- 嵌入与重排（BGE 系列）  
- 向量库（pgvector / Qdrant）  
- Java 网关封装（Spring AI / LangChain4j）  
- 评估系统（Ragas）

---

## 3. 8 周落地计划（按周拆解）

### Week 1：选定场景 + Dify 跑通最小 Demo

**目标：**  
- 选定业务场景  
- 10 分钟跑出一个能问答文档的 Demo

**任务：**
1. 选定主线场景（推荐：企业知识助手）
2. 整理 30–50 份文档（产品文档/接口文档/FAQ）
3. 在 Dify：
   - 创建知识库并导入文档  
   - 用知识库问答组件跑通  
   - 搭建一个简单工作流（输入 → 检索 → 回答）

**产出：**
- 可操作的知识库 Demo  
- 文档答问效果初步体验

---

### Week 2：构建评估集与基线质量评估

**目标：**  
用“数据”替代“感觉”，建立真实评估体系。

**任务：**
1. 收集 20–30 条真实问题  
2. 创建评估集（问题 + 标准答案 + 参考文档）
3. 使用 Ragas 等工具进行初评  
4. 调整 RAG 基础参数（chunk、TopK）

**产出：**
- 基线评估表（Excel/Markdown）  
- 一份“RAG 初始化报告”
  - 正确率  
  - 延时  
  - 弱项分类

---

### Week 3：RAG 微服务（代码化） + Java 网关封装

**目标：**  
从平台 Demo 迈入“可维护的工程后端”。

**任务：**
1. 使用 Python + FastAPI 实现 RAG 微服务：  
   - `/qa` 接口  
   - 嵌入、chunk、向量检索  
2. 使用 pgvector/Qdrant 存储向量  
3. 使用 LlamaIndex 或 LangChain 重建 RAG 流程  
4. 在 Java 构建 API 网关（Spring Boot + Spring AI）

**产出：**
- 可部署的 RAG API 服务  
- Java 网关服务  
- 简单架构图 + 部署说明

---

### Week 4：实现 Agent（ReAct + 工具调用）+ 可视化轨迹

**目标：**  
升级为“能调用工具”的智能体，支持多步骤推理。

**任务：**
1. 设计 2–3 个工具（Tool）  
   - `search_docs`  
   - `query_db`  
   - `send_notice`
2. 基于 LangGraph 或自研简版 ReAct 实现 Agent  
3. 解析 Thought → Action → Observation  
4. 前端或日志中实现“Agent 执行轨迹”可视化

**产出：**
- 可用的 Agent API  
- 工具调用完整链路  
- 前端展示 Agent 的推理步骤

---

### Week 5：权限、安全、审计体系

**目标：**  
让智能体“安全可控”。

**任务：**
1. Java 网关实现简单 RBAC  
2. 敏感 Tool 加入人工确认（例如前端弹窗）  
3. 全链路审计日志：  
   - 用户  
   - 问题  
   - Agent 思考过程  
   - 工具调用  
   - 结果

**产出：**
- RBAC + 审批系统  
- 可追溯的完整审计数据

---

### Week 6：检索优化与知识库治理

**目标：**  
显著提高问答质量。

**任务：**
1. 引入 Hybrid（向量 + 关键词）检索  
2. 引入重排模型（BGE reranker）  
3. 对知识库做“结构化治理”：  
   - 高价值摘取  
   - 数据清洗  
   - 统一标题/段落格式  
4. 使用 Week 2 的评估集进行对比测试

**产出：**
- RAG 升级版  
- “优化前后”指标图表  
- 新的数据结构/数据规范

---

### Week 7：模型切换（云/本地）+ 运营面板

**目标：**  
具备真实生产可用的“弹性与可观测性”。

**任务：**
1. 抽象模型调用层（统一接口）  
2. 支持云模型与本地小模型切换  
3. 构建运营统计：  
   - 调用量  
   - 成本估算  
   - 延时  
   - 常见失败问题  
4. 可视化展示（Grafana / 前端界面）

**产出：**
- 模型切换架构  
- 运营监控面板  
- 成本评估表

---

### Week 8：内部试点 + 复盘总结

**目标：**  
进入真实业务试用与总结阶段。

**任务：**
1. 在 10–30 人小团队开展试点  
2. 收集使用数据与主观评价  
3. 记录问题、优化建议  
4. 输出复盘报告：  
   - 架构  
   - 指标  
   - 风险  
   - 业务收益  
   - 下一步规划

**产出：**
- 内部试点版本（可用）  
- 可向领导展示的复盘 PPT/文档  
- 一套可复用落地方法论

---

## 4. 技术栈（最小可行 BOM）

### 4.1 平台层
- Dify：快速搭建知识库问答/Agent 工作流  
- 或其他低代码平台

### 4.2 模型层
- Embedding：BGE / m3  
- Reranker：BGE reranker

### 4.3 向量库
- pgvector（推荐，轻成本）  
- Qdrant（需要时升级）

### 4.4 后端与 Agent 层
- Python + FastAPI  
- LlamaIndex / LangGraph

### 4.5 Java 接入
- Spring Boot  
- Spring AI / LangChain4j

### 4.6 评估与观测
- Ragas  
- 数据库 + 简单报表或 Grafana

---

## 5. 心法与高阶建议

1. 从场景出发，而不是技术出发  
2. 用平台快速出样，用代码做工程化升级  
3. 评估要从第 2 周开始就引入  
4. Agent 要可视化、可控、可回滚  
5. 用熟悉的基础设施，降低复杂度和运维成本  
6. 持续补强 AI 辅助写代码，提高开发效率
